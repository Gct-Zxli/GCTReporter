## 📊 第一部分：Epic概述分析

### 1.1 Epic的业务背景与核心价值

#### 项目背景

开发一个**面向程序员的报表生成工具**，支持三端架构（管理端、设计端、用户端），使得开发人员能够快速创建、分享和管理SQL查询报表。这是对商业报表工具（如帆软、润乾）的核心功能子集化实现。

#### 核心价值主张

| 维度 | 描述 |
|------|------|
| **用户痛点** | 开发人员需要快速创建和共享SQL查询结果，但现有工具部署复杂、学习成本高 |
| **解决方案** | 提供轻量级报表平台，支持参数化查询、权限控制、Excel导出 |
| **商业价值** | 降低报表开发成本，提高数据共享效率，支持多角色协作 |
| **竞争优势** | 专为程序员设计，学习成本低，快速部署，深度集成开发工具链 |

#### SMART目标分解

**具体性（Specific）**
- 开发支持三端架构的报表生成工具
- 实现SQL查询管理、报表模板设计、权限控制三大核心功能
- 支持角色-报表权限关联和多用户协作

**可衡量性（Measurable）**
- 完成16人日核心功能开发
- 通过40项P0级别验收标准
- 性能指标：查询<3秒、导出<5秒、列表加载<1秒
- 零P0级Bug（无数据泄露、系统崩溃、SQL注入）

**可实现性（Achievable）**
- 团队规模：2名资深开发人员，熟悉SpringBoot和Vue.js
- 技术栈成熟，无复杂算法依赖
- MVP范围明确，不包含复杂图表、拖拽设计器、树形权限

**相关性（Relevant）**
- 直接解决程序员群体的报表共享需求
- 符合企业数据民主化趋势
- 为后续商业化版本打基础

**时间限制（Time-bound）**
- MVP交付周期：2周（10个工作日）
- 工作量：20人日（16核心+4Buffer）
- 预计风险承载度：20%

##### 角色1：系统管理员（Admin）

```yaml
角色特征:
```

```yaml
  1. 输入SQL语句，定义查询参数
```

```yaml

  - 身份: 业务人员、决策者、一线员工
```

## 📋 第二部分：Epic功能拆解分析

### 2.1 核心功能模块（MVP必须）

#### 模块A：SQL查询管理（设计端核心）

```yaml
功能模块: SQL查询管理 ⭐⭐⭐
功能描述:
  - 支持输入SQL查询语句（仅SELECT）
  - 定义查询参数（参数名、类型、默认值、必填性）
  - 支持参数占位符格式: #{paramName}
  - 测试执行查询，验证结果

技术难点:
  1. SQL安全校验（防止注入攻击）
     - 黑名单关键字检查（DROP/DELETE/TRUNCATE等）
     - 仅允许SELECT语句
  2. PreparedStatement参数化
     - 参数绑定，防止注入
  3. 查询超时控制（5秒）
     - 防止大表笛卡尔积导致系统hung

工作量: 2人日
依赖: 无
关键用户故事:
  - US-01: 作为设计者，我想输入SQL语句并定义参数
  - US-02: 作为设计者，我想验证SQL是否存在安全风险
  - US-03: 作为设计者，我想执行查询并预览结果（测试）

验收标准:
  - ✅ 支持${paramName}占位符语法
  - ✅ 检测出常见注入攻击（DROP/DELETE等）
  - ✅ 5秒超时自动中止查询
  - ✅ 参数类型支持：String/Integer/Date/Decimal
  - ✅ 返回JSON格式的列元数据和数据行
```

#### 模块B：报表模板设计（设计端核心）

```yaml
功能模块: 报表模板设计 ⭐⭐⭐
功能描述:
  - 基于SQL查询结果，配置列展现属性
  - 支持自定义字段：显示名称、列宽、数据格式
  - 存储报表元信息（名称、描述、创建人）
  - 列配置持久化为JSON结构

技术难点:
  1. 动态列提取
     - 从ResultSetMetaData获取列信息
     - 自动生成初始列配置
  2. 列配置反序列化
     - JSON配置灵活性 vs 查询性能
  3. 格式化处理
     - 日期格式(yyyy-MM-dd vs dd/MM/yyyy)
     - 数字千位符与精度
     - 货币符号

工作量: 2人日
依赖: SQL查询管理
关键用户故事:
  - US-04: 作为设计者，我想自定义报表列的显示名称
  - US-05: 作为设计者，我想设置列宽和数据格式
  - US-06: 作为设计者，我想保存报表模板供后续使用

验收标准:
  - ✅ 支持格式化类型：日期/数字/货币
  - ✅ 列宽范围：50px-500px
  - ✅ 支持隐藏不显示的列（逻辑删除）
  - ✅ 列顺序可调整
  - ✅ 报表元信息包含名称/描述/创建人/创建时间
```

#### 模块C：报表预览（设计端独享）

```yaml
功能模块: 报表预览 ⭐⭐⭐
所属端: 设计端（Designer端专属）
功能描述:
  - **定位**: 设计阶段的测试验证功能
  - **用户**: 设计者在创建报表时，需要多次验证SQL和参数配置是否正确
  - 前端表格展示查询结果
  - 分页显示（每页100条）
  - 按列排序，实时参数输入
  - 支持多次调试迭代

技术难点:
  1. Vue动态列渲染
     - Element Table with v-for动态列
     - 列顺序变化重新渲染
  2. 前端分页与排序
     - 大数据集（超1000行）的分页策略
     - 排序是否需要后端参与
  3. 参数表单自动生成
     - 根据参数定义动态生成输入控件
     - 参数类型映射：String->Text, Date->Datepicker等

工作量: 2人日
依赖: 报表模板设计
关键用户故事:
  - US-07: 作为设计者，我想在设计阶段预览报表数据
  - US-08: 作为设计者，我想修改参数并重新执行查询
  - US-09: 作为设计者，我想验证列宽和格式化是否符合预期

验收标准:
  - ✅ 显示查询结果表格（带列名和数据）
  - ✅ 每页100条记录的前端分页
  - ✅ 支持列头点击排序
  - ✅ 参数输入表单自动生成
  - ✅ 重新执行查询无需刷新页面
  - ✅ 仅ADMIN和DESIGNER角色可见
```

#### 模块D：报表查询（用户端核心）

```yaml
功能模块: 报表查询 ⭐⭐⭐
所属端: 用户端（Viewer端专属）
功能描述:
  - **定位**: 普通用户的日常工作功能
  - **用户**: 业务人员、决策者查询自己有权限的报表
  - 查看授权报表列表（权限过滤）
  - 选择特定报表进入查询页面
  - 输入查询参数
  - 执行查询并展示结果（带分页、排序）

技术难点:
  1. 权限过滤
     - 仅返回当前用户角色有权限的报表
     - 路由守卫：防止直接访问无权报表
  2. 参数表单自动生成
     - 同模块C，重用参数解析逻辑
  3. 查询结果渲染
     - 复用模块C的表格渲染逻辑
     - 但增加Excel导出按钮

工作量: 2人日
依赖: 报表模板设计、权限控制
关键用户故事:
  - US-10: 作为普通用户，我想查看自己有权限的报表列表
  - US-11: 作为普通用户，我想选择报表并输入参数执行查询
  - US-12: 作为普通用户，我想查看排序和分页的结果

验收标准:
  - ✅ 报表列表仅显示已授权的报表
  - ✅ 参数表单与设计端保持一致
  - ✅ 查询结果支持分页、排序
  - ✅ 无权访问的报表返回403 Forbidden
  - ✅ 查询失败显示友好错误提示
```

#### 模块E：Excel导出（用户端增强）

```yaml
功能模块: Excel导出 ⭐⭐
所属端: 用户端（Viewer端使用）
功能描述:
  - 导出查询结果为Excel文件（.xlsx格式）

工作量: 1人日
依赖: 报表查询
关键用户故事:
  - US-13: 作为普通用户，我想导出报表结果为Excel

验收标准:
  - ✅ 生成合法的.xlsx文件
  - ✅ 文件名格式正确
  - ✅ 表头和数据行完整
  - ✅ 超过5000行提示：请优化查询条件
  - ✅ 导出时间<5秒
```

#### 模块F：权限控制（管理端核心）

```yaml
功能模块: 基于角色的权限控制（RBAC） ⭐⭐⭐
所属端: 管理端（用户-角色）、设计端（报表-角色）、全端（权限校验）
功能描述:
  - 定义三个固定角色：Admin、Designer、Viewer
  - 用户-角色关联（一用户多角色）
  - 报表-角色关联（一报表对应多个角色）
  - 权限继承链：Admin > Designer > Viewer
  - 拦截器校验用户角色与报表权限

权限矩阵:
  管理员(ADMIN):
    - 管理端: 全部功能（用户/角色/权限）
    - 设计端: 全部功能（报表预览）
    - 用户端: 全部功能（报表查询）
    - 权限: 无限制
  
  设计者(DESIGNER):
    - 管理端: 无
    - 设计端: 创建/编辑/授权/预览报表
    - 用户端: 查看授权报表（报表查询）
    - 权限: 仅显示分配给DESIGNER角色的报表
  
  普通用户(VIEWER):
    - 管理端: 无
    - 设计端: 无
    - 用户端: 查看授权报表（报表查询）
    - 权限: 仅显示分配给VIEWER角色的报表

技术难点:
  1. 多对多关联表
     - user_role表
     - report_role表
     - 权限查询优化（避免N+1）
  2. 权限缓存
     - 用户登录后缓存其角色和权限报表列表
  3. 权限继承实现
     - Admin查询时自动返回所有报表

工作量: 2人日
依赖: 用户认证
关键用户故事:
  - US-15: 作为管理员，我想为用户分配角色
  - US-16: 作为管理员，我想定义报表对应的角色
  - US-17: 作为系统，我想在API层校验用户权限

验收标准:
  - ✅ 支持用户分配多个角色
  - ✅ 支持报表分配给多个角色
  - ✅ 权限继承正确（Admin可访问所有）
  - ✅ 无权报表返回403
  - ✅ 权限校验不影响查询性能（<100ms）
```

### 2.3 拆解策略与用户故事规划

#### 拆解维度选择

| 拆解维度 | 优点 | 缺点 | 选择 |
|---------|------|------|------|
| **按角色** | 符合业务流，易于验收 | 故事粒度难以控制 | ✅ 采用 |
| **按功能** | 技术实现清晰 | 验收标准不明确 | ✅ 结合 |
| **按业务流** | 故事独立完整 | 跨端分布，难以排期 | ⚠️ 部分采用 |
| **按技术栈** | 易于分工 | 故事价值不清晰 | ❌ 不采用 |

#### 优选拆解方案

**分层拆解**：先按角色分流（3条线），再按功能细分

```
第一层：按角色分流
├─ 管理员线：用户/角色管理 (2个故事)
├─ 设计者线：SQL+模板设计+预览 (5个故事)
└─ 用户线：查询+导出 (3个故事)

第二层：按功能细分
└─ 每条线内按功能优先级排序
```

#### 预期用户故事数量

```
核心功能故事: 12-15个
├─ 设计端故事: 6-7个
├─ 用户端故事: 3-4个
├─ 管理端故事: 2-3个
└─ 全端故事: 2个
```

## 🎯 第三部分：拆解风险与依赖分析

### 3.1 技术风险评估

#### 风险1：SQL注入安全（高风险）

```yaml
风险等级: 🔴 高（P0）
风险描述: 
  用户输入恶意SQL可能导致数据泄露、破坏、性能攻击

影响范围:
  - 模块A (SQL查询管理)
  - 可能导致整个项目无法上线

缓解方案:
  1. 黑名单关键字检查 (DROP/DELETE/TRUNCATE等)
  2. 仅允许SELECT语句
  3. 专用只读数据库账号 (仅SELECT权限)
  4. PreparedStatement参数化
  5. 5秒查询超时限制

工作量影响: +0.5人日 (安全校验)
责任人: 后端核心开发

建议行动:
  - US-02需要详细的验收标准
  - 添加安全测试故事（尝试注入攻击）
```

#### 风险2：大数据量导出性能（中风险）

```yaml
风险等级: 🟡 中（P1）
风险描述:
  导出5000行Excel文件可能导致内存溢出或超时

影响范围:
  - 模块E (Excel导出)
  - 用户体验降级

缓解方案:
  1. 硬限5000行导出上限
  2. 流式写入Excel (避免全量加载内存)
  3. 异步导出 (生成后台任务，用户查询结果)
  4. 提示用户优化查询条件

工作量影响: +0.5人日 (异步处理)
责任人: 后端核心开发

建议行动:
  - MVP版本先采用同步导出+5000行限制
  - V2版本升级为异步导出+后台队列
```

#### 风险3：多端权限同步（中风险）

```yaml
风险等级: 🟡 中（P0）
风险描述:
  三端权限校验逻辑分散，容易出现权限漏洞

影响范围:
  - 模块F (RBAC)
  - 设计端+用户端权限不一致

缓解方案:
  1. 后端权限校验为单一真源 (拦截器)
  2. 前端权限缓存，登录时同步一次
  3. 权限变更需要用户重新登录
  4. 添加权限审计日志

工作量影响: +1人日 (拦截器 + 前端缓存)
责任人: 全栈团队

建议行动:
  - 专门的技术故事：实现权限拦截器
  - 添加集成测试验证权限边界
```

```
SQL查询   ✅       ⬅️       ⬅️      ⬅️      ⬅️      ⬅️

说明:
  ✅ = 无依赖
  ⬅️ = 依赖该模块
  ➡️ = 被该模块依赖
```
